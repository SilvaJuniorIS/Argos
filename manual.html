Produto

O que o ARGOS faz

MVP comercial focado no menor fluxo funcional.

1. Cadastra processos Recebe objeto, secretaria, justificativa, modalidade, prazo e itens.
2. Gera minutas Cria ETP ou Termo de Referencia com prompts estruturados.
3. Permite revisao Entrega um editor simples para ajuste humano antes do uso oficial.
4. Exporta DOCX Gera arquivo apresentavel com cabecalho institucional.
Arquitetura

Como o sistema esta organizado

O MVP separa frontend, backend, prompts, templates, exports e documentacao. O backend concentra regras de negocio, banco SQLite, geracao DOCX e chamadas de IA via LLM Gateway. O frontend consome somente a API FastAPI e nunca recebe chaves sensiveis.

Operacao

Fluxo recomendado de uso

1. Abrir dashboard Veja totais, status e ultimos processos cadastrados.
2. Criar processo Preencha dados gerais e itens do processo.
3. Gerar minuta Escolha ETP ou TR e aguarde a IA estruturar o texto.
4. Revisar conteudo Leia, ajuste e salve a revisao humana.
5. Exportar DOCX Baixe o documento para tramitacao interna.
Imagem 2: cadastro guiado do processo.

Boas praticas no cadastro

  • Escreva o objeto com clareza, evitando descricoes genericas.
  • Informe a area requisitante exatamente como deve aparecer na minuta.
  • Use a justificativa para explicar a necessidade publica e o problema a resolver.
  • Cadastre todos os itens relevantes, com quantidade e unidade de medida.
  • Use observacoes para restricoes, prazos, locais de entrega ou detalhes tecnicos.
Documentos

ETP e Termo de Referencia

Documento Quando usar Principais secoes geradas
ETP Planejamento inicial da contratacao e estudo da necessidade. Necessidade, requisitos, mercado, solucao escolhida, quantidades, valor, riscos e conclusao.
Termo de Referencia Detalhamento da contratacao para orientar licitacao, execucao e fiscalizacao. Objeto, justificativa, especificacoes, quantitativos, gestao, pagamento, obrigacoes e sancoes.
Importante: todo documento gerado e uma minuta de apoio. Antes do uso oficial, faca revisao tecnica, juridica, orcamentaria e administrativa.
LLM Gateway

Configuracao de IA

O backend usa uma camada chamada LLM Gateway. Ela escolhe o provedor principal, tenta fallback quando necessario e normaliza a resposta para o restante do sistema. Nenhuma tela do frontend recebe chaves de IA.

OpenRouter como principal Configuracao recomendada para o MVP por ser compativel com Chat Completions.
LLM_PROVIDER=openrouter LLM_FALLBACK_PROVIDER=openai LLM_TIMEOUT_SECONDS=45 OPENROUTER_API_KEY=sk-or-sua-chave OPENROUTER_BASE_URL=https://openrouter.ai/api/v1 OPENROUTER_DEFAULT_MODEL=openrouter/auto OPENROUTER_HTTP_REFERER=http://localhost:5174 OPENROUTER_X_TITLE=ARGOS OPENAI_API_KEY=sk-sua-chave-opcional OPENAI_DEFAULT_MODEL=gpt-4o-mini
OpenAI como unico provedor Use se quiser operar sem OpenRouter no momento.
LLM_PROVIDER=openai LLM_FALLBACK_PROVIDER= OPENAI_API_KEY=sk-sua-chave OPENAI_DEFAULT_MODEL=gpt-4o-mini OPENAI_TIMEOUT_SECONDS=45
Variavel Funcao Exemplo
LLM_PROVIDER Define o provedor principal. openrouter
LLM_FALLBACK_PROVIDER Define o provedor usado se o principal falhar. openai
OPENROUTER_DEFAULT_MODEL Modelo padrao no OpenRouter. openrouter/auto
OPENAI_DEFAULT_MODEL Modelo padrao no provedor OpenAI. gpt-4o-mini
Exportacao

Revisao e DOCX

Depois que a IA gera a minuta, o usuario entra na tela de revisao. O conteudo pode ser editado diretamente no navegador, salvo como revisado e exportado em DOCX.

  • Use o editor para corrigir trechos genericos ou incompletos.
  • Substitua campos "A preencher" por dados reais do processo.
  • Confirme fundamentacao legal, prazos, quantidades e obrigacoes.
  • Exporte somente depois de validar a minuta com a equipe responsavel.

Producao

Deploy no Render

Backend Use o servico `argos-api` definido no `render.yaml`.
Root Directory: backend Build Command: pip install -r requirements.txt Start Command: uvicorn app.main:app --host 0.0.0.0 --port $PORT DATABASE_URL=sqlite:////var/data/argos.db
Frontend Use o servico static `argos-web` definido no `render.yaml`.
Root Directory: frontend/fiscalbot-web Build Command: npm ci && npm run build Publish Directory: dist VITE_API_URL=https://sua-url-backend.onrender.com
No Render, SQLite com disco persistente exige Web Service com disco configurado. Para uso multiusuario ou volume maior, o proximo passo recomendado e migrar para PostgreSQL.
Suporte rapido

Problemas comuns e solucoes

Sintoma Causa provavel Como resolver
Dashboard nao carrega Backend desligado ou URL incorreta. Confirme `uvicorn` em `localhost:8000` e `VITE_API_URL` no frontend.
Erro ao gerar documento Chave IA ausente, invalida ou provedor indisponivel. Revise `OPENROUTER_API_KEY`, `OPENAI_API_KEY`, `LLM_PROVIDER` e logs do backend.
CORS bloqueando requisicoes Frontend fora da lista permitida. Inclua a URL correta em `CORS_ORIGINS`.
DOCX nao baixa Documento inexistente ou erro em `python-docx`. Confira se o documento foi salvo e se as dependencias foram instaladas.
Deploy perde dados SQLite sem disco persistente. Use disco em `/var/data` ou migre para PostgreSQL.

  • Frontend abrindo em `/argos`.
  • `OPENROUTER_API_KEY` configurada no backend.
  • Fallback OpenAI configurado, se disponivel.
  • Um processo de exemplo ja cadastrado.
  • Geracao de ETP testada antes da reuniao.
  • Exportacao DOCX validada.
  • Aviso de revisao humana explicado ao cliente.
Roteiro sugerido: comece pelo dashboard, crie um novo processo simples, gere uma minuta, edite um paragrafo no navegador e exporte o DOCX. Esse caminho mostra valor real sem depender de funcionalidades futuras.
